Collection Description
This project demonstrates how to design and execute automated API tests using Postman and Newman. We are testing the GoRest Users API with the base URL: "[https://gorest.co.in/public/v2/users"](https://gorest.co.in/public/v2/users) This endpoint allows CRUD operations on user data and serves as a good candidate for showcasing end-to-end API testing. (1) Tools & Setup - Postman → for designing requests and writing test scripts. - Newman → for running collections in CI/CD pipelines. - GitHub → for version control and collaboration. - Jenkins → for automated runs & reporting. (2)Test Scenarios We created tests around the key CRUD endpoints: (i) Create User (POST /users) - Verify status code 201. - Validate response schema (fields: id, name, email, gender, status). - Save id for subsequent tests. (ii) Get Users (GET /users) - Verify status code 200. - Assert response contains an array. - Validate specific fields (id, email) are present. (iii) Get User by ID (GET /users/{{userID}}) - Verify status code 200. - Assert id matches the created user. (iv) Update User (PUT /users/{{userID}}) - Verify status code 200. - Confirm updated fields (name, status) are reflected. (v) Delete User (DELETE /users/{{userID}}) - Verify status code 204. - Assert a subsequent GET returns 404. (3) Test Automation with Newman \-> newman run gorest_collection.json -e gorest_environment.json \--reporters cli,html --reporter-html-export newman-report.html (4) CI/CT Integration - Added collection & environment files into a GitHub repo. - Created a Jenkins pipeline to: - Run the Newman command. - Archive & email reports. - Schedule runs (hourly or 2x daily). (6) Results - Automated API tests cover Create → Read → Update → Delete flows. - Reports show pass/fail status of all requests. - Scheduling ensures continuous validation of API availability and functionality.
